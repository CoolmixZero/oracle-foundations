- ## čo znamena domenova integrita tabulky?
**Доменная целостность таблицы в Oracle SQL означает, что значения, хранящиеся в столбцах таблицы, должны соответствовать определенному набору правил и ограничений, называемых доменом. Домен определяет допустимый диапазон значений, тип данных и другие свойства столбца.**

Когда говорят о доменной целостности, это означает, что значения в каждом столбце таблицы должны соответствовать определенному домену или правилам, связанным с этим столбцом. Например, если у столбца есть ограничение типа данных INTEGER, то только целочисленные значения могут быть вставлены в этот столбец. Если попытаться вставить значение с плавающей запятой или символьное значение, будет нарушена доменная целостность.

Доменная целостность обеспечивает правильное и согласованное хранение данных в таблице, гарантируя, что значения соответствуют ожидаемым требованиям и типам данных. Если данные нарушают правила доменной целостности, то могут возникать ошибки, и база данных не позволит вставить или изменить такие значения, чтобы поддерживать целостность данных.

- ## Popište 2. normalovu formu.
Вторая нормальная форма (2NF) является одной из стандартных форм нормализации в реляционной модели данных. Она гарантирует отсутствие повторяющихся групп данных внутри таблицы.

**Вторая нормальная форма (2NF) требует, чтобы таблица имела следующие два свойства:**
- **Быть в 1NF: Таблица должна быть в первой нормальной форме (1NF), что означает, что каждая ячейка в таблице должна содержать только атомарные значения, а каждый столбец должен иметь уникальное имя.**
- **Устранение частичных зависимостей: Все неключевые столбцы должны полностью зависеть от каждого ключевого столбца. Если в таблице есть составной ключ (ключ, состоящий из нескольких столбцов), то каждый неключевой столбец должен зависеть от всего составного ключа, а не от его части.**

Вторая нормальная форма (2NF) помогает избежать избыточности данных и повышает эффективность запросов к базе данных. Нормализация до 2NF обычно выполняется путем разделения таблицы на две или более таблиц и установления связей между ними на основе зависимостей между ключевыми и неключевыми столбцами.

- ## čo znamena referenčna integrita tabulky?
**Референциальная целостность таблицы в Oracle SQL означает, что связи между таблицами должны быть соблюдены и не нарушены. Это включает в себя установление ограничений на внешние ключи, которые обеспечивают согласованность данных между связанными таблицами.**

Когда говорят о референциальной целостности, это означает, что значения в столбце, являющемся внешним ключом, должны ссылаться на существующие значения в столбце родительской таблицы (связанной таблицы). Это гарантирует, что данные, связанные между таблицами, остаются согласованными и корректными.

Ограничения референциальной целостности включают:

1. Ограничение внешнего ключа (FOREIGN KEY CONSTRAINT): Это ограничение определяет, что значения в столбце внешнего ключа должны ссылаться на существующие значения в столбце родительской таблицы. Таким образом, запрещается вставка, обновление или удаление данных, которые нарушают связи между таблицами.

2. Ограничение удаления (DELETE CONSTRAINT): Это ограничение определяет, как обрабатывать данные, когда происходит удаление записи из родительской таблицы. Например, можно задать ограничение, чтобы при удалении записи из родительской таблицы автоматически удалялись все связанные записи в дочерних таблицах или чтобы запрещалось удаление, если существуют связанные записи.

Референциальная целостность обеспечивает целостность данных в базе данных и предотвращает возможные аномалии, такие как некорректные ссылки или орфанские записи. Это помогает поддерживать согласованность данных и обеспечивает надежность операций в базе данных.

- ## Popište obmedzenie FOREIGN KEY.
Ограничение FOREIGN KEY (внешний ключ) в Oracle SQL используется для определения связей между таблицами и обеспечения референциальной целостности данных. Оно определяет, что значения в столбце (или столбцах) внешнего ключа в таблице-потомке должны ссылаться на существующие значения в столбце (или столбцах) родительской таблицы.

Вот основные аспекты ограничения FOREIGN KEY:

1. Определение внешнего ключа: Внешний ключ определяется в таблице-потомке (дочерней таблице) и указывает на столбец (или столбцы), который ссылается на столбец (или столбцы) родительской таблицы. Столбец внешнего ключа содержит значения, которые должны существовать в столбце родительской таблицы.

2. Связь между таблицами: Ограничение FOREIGN KEY устанавливает связь между таблицами. Значение внешнего ключа должно быть равным значению первичного ключа или уникального ключа в родительской таблице. Таким образом, создается связь "один-ко-многим" или "один-к-одному" между таблицами.

3. Обработка операций вставки, обновления и удаления: Ограничение FOREIGN KEY определяет действия при вставке, обновлении или удалении данных, которые могут нарушить референциальную целостность. Варианты действий включают автоматическое удаление (CASCADE), установку NULL (SET NULL), установку значения по умолчанию (SET DEFAULT) или отказ в операции (NO ACTION).

4. Проверка данных: При нарушении ограничения FOREIGN KEY база данных не позволит вставлять, обновлять или удалять данные, которые нарушают связи между таблицами. Это помогает поддерживать целостность данных и избегать ошибок или некорректных ссылок.

Ограничение FOREIGN KEY является мощным инструментом для установления связей между таблицами и обеспечения согласованности данных. Оно позволяет создавать сложные структуры баз данных, в которых данные в разных таблицах тесно связаны между собой.

- ## čo znamena 3-hodnotova logika v Oracle SQL jazyku?
**Трехзначная логика в языке Oracle SQL относится к концепции, в которой логические выражения могут принимать одно из трех возможных значений: истина (TRUE), ложь (FALSE) или неизвестно (UNKNOWN).**

В отличие от двухзначной логики, где выражения могут быть только истинными или ложными, трехзначная логика вводит третье значение "неизвестно". Это возникает, когда результат логического выражения не может быть определен однозначно из-за отсутствия информации или недостатка данных для принятия окончательного решения.

В трехзначной логике в Oracle SQL выполняются следующие правила:

1. Истина (TRUE):

- TRUE AND TRUE = TRUE
- TRUE OR FALSE = TRUE
- NOT FALSE = TRUE
2. Ложь (FALSE):

- FALSE AND FALSE = FALSE
- FALSE OR TRUE = TRUE
- NOT TRUE = FALSE
3. Неизвестно (UNKNOWN):

- TRUE AND UNKNOWN = UNKNOWN
- UNKNOWN AND FALSE = FALSE
- UNKNOWN OR TRUE = TRUE
- UNKNOWN OR FALSE = UNKNOWN
- NOT UNKNOWN = UNKNOWN

Неизвестное значение (UNKNOWN) обычно возникает в следующих ситуациях:

- Когда один или оба операнта логического выражения содержат NULL (неопределенное значение).
- Когда используются некоторые специальные операции, такие как сравнение с NULL или операции с нулевым делителем.

Трехзначная логика в Oracle SQL обеспечивает более гибкий способ работы с логическими выражениями и позволяет корректно обрабатывать неопределенные значения или отсутствие информации в данных.

**Различие между представлениями ALL_OBJECTS и USER_OBJECTS заключается в уровне доступности объектов в базе данных. ALL_OBJECTS предоставляет обзор всех объектов, на которые пользователь имеет привилегии, включая объекты, принадлежащие другим пользователям, а USER_OBJECTS ограничено только объектами, принадлежащими текущему пользователю.**

- ## Akym prikazom vieme spojiť riadky 2 tabuliek, ktore maju rovnaku hodnotu kluča?
Для объединения строк двух таблиц, имеющих одинаковое значение ключа, в Oracle SQL используется оператор **JOIN**.

Оператор JOIN позволяет соединять строки из двух или более таблиц на основе условия соответствия значений ключа.

- ## čo je vnoreny dopyt?
**Вложенный запрос (также известный как вложенный или внутренний запрос) в языке SQL представляет собой запрос, который включен внутри другого запроса.** Внутренний запрос выполняется с использованием результатов внешнего запроса и может использоваться для получения дополнительных данных или фильтрации результатов.

Внутренний запрос может быть использован в различных частях SQL запроса, включая SELECT, FROM, WHERE, HAVING и других.

Внутренний запрос может быть использован для выполнения различных операций, таких как фильтрация данных, поиск совпадающих значений, агрегация данных, проверка условий и многое другое. Он позволяет более гибко и точно управлять результатами запроса, используя данные из других таблиц или подзапросов.

Однако следует быть осторожным с использованием вложенных запросов, так как они могут иметь влияние на производительность запроса. В некоторых случаях может быть эффективнее использовать соединения таблиц или другие методы для достижения требуемых результатов.

- ## čo je to pohlad?
**В Oracle SQL понятие "представление" (view) используется для создания виртуальной таблицы, которая является результатом выполнения запроса. Представление представляет собой логически структурированные данные, которые могут быть использованы для упрощения доступа и анализа данных.**

Представление определяет набор столбцов и их типы данных, а также определяет правила, по которым данные из одной или нескольких таблиц объединяются и фильтруются. Представление сохраняется в базе данных и может быть использовано повторно для выполнения запросов без необходимости написания сложных запросов снова.

Представление предоставляет следующие преимущества:
- Упрощает доступ к данным, скрывая сложные запросы или операции объединения таблиц.
- Позволяет определить пользовательские виды данных для различных пользователей или приложений.
- Повышает безопасность, так как можно предоставить доступ только к определенным столбцам или строкам данных.
- Улучшает производительность, так как представления могут быть предварительно вычислены и кэшированы.
- 
Представления являются мощным инструментом в Oracle SQL, позволяющим управлять данными и предоставлять пользователю гибкий и удобный способ работы с информацией из базы данных.

- ## čo predstavuje konceptualna uroveň architektury databazobeho systemu?
**Концептуальный уровень в архитектуре базы данных представляет собой уровень, который описывает общую структуру и организацию данных в базе данных, независимо от конкретных деталей физической реализации. Он фокусируется на концептуальной модели данных, которая отражает бизнес-предметную область и ее отношения.**

На концептуальном уровне определяются основные сущности (таблицы), их атрибуты и связи между ними. Концептуальная модель данных является абстракцией реального мира, которую можно представить в виде диаграммы сущность-связь (Entity-Relationship Diagram, ERD).

Некоторые ключевые аспекты концептуального уровня включают:

1. Сущности (Entities): Представляют основные объекты или сущности в предметной области. Например, в базе данных для учета сотрудников могут быть определены сущности "Сотрудник", "Отдел" и "Проект".

2. Атрибуты (Attributes): Описывают характеристики или свойства сущностей. Например, для сущности "Сотрудник" атрибутами могут быть "Имя", "Фамилия", "Дата рождения" и т. д.

3. Связи (Relationships): Определяют отношения между сущностями. Например, связь "Работает в" может быть установлена между сущностями "Сотрудник" и "Отдел", чтобы указать, в каком отделе работает сотрудник.

4. Ограничения целостности (Integrity Constraints): Определяют правила и ограничения для данных на концептуальном уровне. Например, можно определить ограничения на значения атрибутов или ограничения, связанные с отношениями между сущностями.

Концептуальный уровень важен для понимания общей структуры данных и логики бизнес-предметной области. Он служит основой для разработки физической и логической реализации базы данных на более низких уровнях архитектуры.

- ## Vymenujte aspoň jeden vzťah použivany v E-R modely?
В ER-модели (модели сущность-связь) могут существовать различные типы отношений между сущностями. Вот некоторые из основных типов отношений в ER-модели:

1. **Один-к-одному (One-to-One)**: Отношение, при котором каждая запись в одной сущности связана с одной записью в другой сущности. Например, один человек может иметь только один паспорт, и каждый паспорт принадлежит только одному человеку.

2. **Один-ко-многим (One-to-Many)**: Отношение, при котором каждая запись в одной сущности может быть связана с несколькими записями в другой сущности. Например, один отдел может иметь несколько сотрудников, но каждый сотрудник принадлежит только одному отделу.

3. **Многие-к-одному (Many-to-One)**: Обратное отношение к отношению "один-ко-многим". Здесь каждая запись в одной сущности связана с одной записью в другой сущности, но каждая запись во второй сущности может быть связана с несколькими записями в первой сущности. Например, несколько сотрудников могут работать в одном отделе, но каждый отдел принадлежит только одной компании.

4. **Многие-ко-многим (Many-to-Many)**: Отношение, при котором каждая запись в одной сущности может быть связана с несколькими записями в другой сущности, и наоборот. Для представления этого типа отношений требуется дополнительная таблица, называемая таблицей-связью или таблицей-пересечением. Например, множество студентов может быть записано на несколько курсов, и каждый курс может иметь несколько студентов.

Кроме того, в ER-модели можно использовать дополнительные типы отношений, такие как отношение включения и ассоциативное отношение. Включение описывает отношение "часть-целое", где одна сущность является частью другой сущности. Ассоциативное отношение используется для представления дополнительной информации или свойств, связанных с отношением между двумя сущностями.
